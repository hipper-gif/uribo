# うりぼー 給与・基本給データの仕組みメモ

> このファイルは「給料設定の基本給はどこから読んでいるのか／時給→月給に変わったとき過去分はどうなるのか」を
> 調べた結果をまとめた備忘録です。あとで読み返す用。（作成: 2026-07-01）

---

## ざっくり結論（ここだけ読めばOK）

- 基本給の読み口は **2つある**。「画面に表示する時」と「毎月の給与レコードを作る時」で、読み方も挙動も違う。
- **過去月の給与の実績データは壊れない。** 毎月その時点の金額を焼き付けて保存しているので、あとで設定を変えても過去は書き換わらない。
- **時給 → 月給に変えるときは、既存の設定を上書きせず「いつから変わったか（日付つき）」で新しく履歴を足す。** これを守れば過去も未来も正しくなる。上書きすると過去まで新しい条件で計算されてしまう危険がある。

---

## 1. 基本給の「正本」は3つの層に分かれている

| 層 | どこが正本か | 中身 |
|---|---|---|
| 等級表（ランク→金額の対応表） | **Mneme の `salary_grades`**（美容部＝department_id=3、時期別に管理） | 雇用形態×ランクごとの基本給マスタ |
| 個人のランク割当・個別調整・履歴 | **うりぼー の `beauty_employee_grade`** | 誰がどのランクか / 個別加算(override) / いつから有効か |
| 現在値のスナップショット | **Mneme の `employees`（base_salary / salary_type / employment_type）** | 上の2つから計算した「今の値」。他システム（SmileyBase等）が読む用 |

- 同期は **「うりぼー → Mneme」の一方向だけ**。
- Mneme に直接給与を入力すると、うりぼー側の履歴や個別調整が消えるので **やらない**。

---

## 2. 基本給を読む口は「2系統」ある

### 系統A：画面に表示するとき（Mneme の"今の値"を直接読む）

- **目標設定の画面（TargetSetting）** … スタッフの人件費をシミュレーションするのに Mneme の基本給を使う。
  - 「時給」の人は `基本給(時給単価) × 勤務時間`、それ以外は月給そのまま。
- **給与計算の画面（Payroll）** … Mneme からはスタッフの**名前だけ**もらう。表示する基本給は系統Bで焼き付けた値。

> ポイント：Mneme の "今の値" は **1個しか持っていない（過去の履歴を持たない）**。

### 系統B：毎月の給与レコードを作るとき（その月の金額を焼き付けて保存）

- 月次バッチ（sync_salonboard.py）が、
  1. その月時点で有効なランク（`beauty_employee_grade`）
  2. その月時点の金額（Mneme の `salary_grades`）
  を引いて計算し、**うりぼーの `beauty_payroll_monthly` テーブルに焼き付けて保存**する。
- **月給の人（有期雇用・正社員）だけ基本給をセット。時給の人（パート）は基本給0で保存**（時給×勤務時間はTKC側で確定するため。画面でも「¥0表示」になる）。
- すでに「確定済／TKC入力済」になっている月は、手入力した部分を上書きしない安全装置あり。

---

## 3. 時給 → 月給 に変えたとき、過去分はどうなる？（本題）

**データによって挙動が真逆なので注意。**

### ① 過去月の給与"実績"（`beauty_payroll_monthly`）→ 壊れない ✅
- 各月を作った時点の金額を焼き付けているので、あとで設定を変えても過去のレコードは書き換わらない。
- 時給時代の月は「基本給0」で残っている。

### ② Mneme の"今の値"（画面がライブで読む方）→ 現在値で丸ごと上書きされる ⚠️
- Mneme は履歴を持たず今の値1個だけ。同期を走らせると新しい月給値に置き換わる。
- そのため：
  - **未来の目標設定は新しい月給で正しく計算される** ✅
  - **過去月を画面で開くと「今＝月給」で表示され、時給時代の実態とズレる** ⚠️
  - → 過去の正しい姿を見たいときは `beauty_payroll_monthly`（焼き付け値）の方を見る。

---

## 4. 正しい切り替え手順（設計上こうすべき）

1. **`beauty_employee_grade` の既存の行を書き換えない。**
   「いつから月給に変わったか（effective_from＝切替日）」で**新しい行を追加**する。時給時代の行は切替日の前日で閉じる。
   - こうすれば、あとから過去月を作り直しても月ごとに正しい金額が引ける。
   - 既存行を上書きすると、過去月まで新しい雇用形態で計算されてしまう。
2. 必要なら等級表（Mneme `salary_grades`）も時期で区切る。
3. 切り替え後に同期（sync_grades_to_mneme.py）を走らせて、Mneme の"今の値"を更新する。

---

## 5. ついでの注意（事故メモ）

- Mneme の `salary_type` と `employment_type` は**決まった言葉（ENUM）しか受け付けない**。
  それ以外の表記で保存しようとすると、MySQLがエラーを出さず**空っぽで保存してしまう**（過去に事故あり）。
- 表記が2系統あってややこしい：
  - 等級表側：`パート・アルバイト / 有期雇用 / 正社員`
  - Mneme社員側：`パート / 契約社員 / 正社員 / 登録ヘルパー`
  - コード内で対応表（マッピング）を持って変換している。
